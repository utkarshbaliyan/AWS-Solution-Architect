## 1. What is EC2?

EC2 (Elastic Compute Cloud) lets you rent virtual machines — called **instances** — in AWS. You choose the OS, CPU, memory, storage, and networking, launch in minutes, and pay only for what you use.

### Why "Elastic"?

You can scale up (bigger instances), scale out (more instances), and scale down — all on demand. No buying servers, no waiting weeks for hardware.

### What EC2 gives you

- Full control of a virtual server (root/admin access)
- Choice of OS (Linux, Windows, macOS)
- Flexible compute, memory, and networking
- Integration with storage (EBS), networking (VPC), and identity (IAM)

> 💡 Recall from the prerequisites: an EC2 instance is a **virtual machine** running on AWS's hypervisor (Nitro). It's also a **server** in the client-server sense.

## 2. The EC2 Building Blocks

To launch an instance you assemble these pieces:

| Component | What it is |
|---|---|
| **AMI** | The template — OS + pre-installed software |
| **Instance Type** | The hardware sizing — vCPU, RAM, network |
| **EBS Volume** | The virtual hard drive |
| **Key Pair** | SSH/RDP credentials for login |
| **Security Group** | A virtual firewall controlling traffic |
| **VPC / Subnet** | The network the instance lives in |
| **IAM Role** | Permissions the instance assumes (optional but recommended) |
| **User Data** | A boot-time startup script (optional) |

```
┌──────────────────────────────────────────────┐
│              EC2 Instance                      │
│  ┌──────────┐  AMI (OS + software)            │
│  │ Compute  │  Instance Type (vCPU + RAM)     │
│  │          │  EBS Volume (storage)           │
│  └──────────┘                                 │
│  Protected by → Security Group                │
│  Lives inside → VPC / Subnet                  │
│  Accessed via → Key Pair                      │
│  Permissions  → IAM Role                      │
└──────────────────────────────────────────────┘
```

## 3. Instance Types and Families

AWS offers hundreds of instance types, grouped into **families** optimized for different workloads.

### Naming convention

Example: **`m5.xlarge`**

```
m       5        .xlarge
│       │           │
family  generation  size
```

- **m** → family (general purpose)
- **5** → generation (newer = better price/performance)
- **xlarge** → size within the family

Sometimes there's an extra letter: `m5d.xlarge` (d = local NVMe SSD), `m5n` (n = enhanced networking), `m6g` (g = AWS Graviton ARM processor).

### The main families (know these for the exam)

| Family | Letter | Optimized for | Use case |
|---|---|---|---|
| **General Purpose** | T, M | Balanced CPU/memory | Web servers, small DBs, dev |
| **Compute Optimized** | C | High CPU | Batch processing, gaming servers, HPC |
| **Memory Optimized** | R, X, Z | Large RAM | In-memory DBs, caches, big data |
| **Storage Optimized** | I, D, H | High disk throughput/IOPS | Data warehouses, NoSQL, big files |
| **Accelerated Computing** | P, G, F, Inf | GPUs / FPGAs | ML training, video, graphics |

> 🎯 **Exam tip:** Memorize the *mapping*. "Memory-intensive database" → R family. "CPU-heavy batch job" → C family. "Machine learning training" → P/G family.

### Burstable instances (T family)

T-family instances (`t2`, `t3`, `t4g`) use a **CPU credit** system:
- They run at a low baseline CPU
- They earn credits when idle
- They spend credits to "burst" above baseline
- Run out of credits → throttled (or charged for unlimited mode)

> 💡 Great for workloads that are usually quiet with occasional spikes (dev boxes, low-traffic sites). Bad for sustained heavy CPU.

## 4. AMIs — Amazon Machine Images

An AMI is the **template** used to launch an instance. It contains the OS, application server, and any pre-installed software.

### Types of AMIs

| Type | Source |
|---|---|
| **AWS-provided** | Amazon Linux, Ubuntu, Windows Server, etc. |
| **AWS Marketplace** | Vendor-built (often paid) — e.g., a hardened Nginx image |
| **Community** | Shared by other AWS users (use with caution) |
| **Custom (your own)** | You configure an instance, then create an AMI from it |

### Why create custom AMIs?

- **Faster launches** — software is baked in, no install scripts at boot
- **Consistency** — every instance starts identical ("golden image")
- **Auto Scaling** — new instances spin up pre-configured

### Key AMI facts

- AMIs are **region-specific** — to use one in another region, copy it
- An AMI is backed by **EBS snapshots** (for EBS-backed AMIs)
- You can control who can use your AMI (private, specific accounts, or public)

```
Configure an instance → Create AMI → Launch identical copies anywhere
```

## 5. Instance Lifecycle

The states an instance moves through:

```
            launch
   ┌──────────────────────►  ┌─────────┐
   │                         │ pending │
   │                         └────┬────┘
   │                              ▼
   │  start                  ┌─────────┐   stop      ┌─────────┐
   │  ┌─────────────────────│ running │────────────►│ stopping│
   │  │                      └────┬────┘             └────┬────┘
   │  │                           │                       ▼
┌──┴──▼───┐                       │ terminate        ┌─────────┐
│ stopped │◄──────────────────────┼──────────────────│ stopped │
└─────────┘                       ▼                  └─────────┘
                            ┌────────────┐
                            │ terminated │ (gone forever)
                            └────────────┘
```

### Stop vs Terminate vs Hibernate

| Action | What happens | EBS data | Billing |
|---|---|---|---|
| **Stop** | Instance shuts down | Root EBS preserved | No compute charge; EBS still billed |
| **Terminate** | Instance deleted | Root EBS deleted (by default) | Nothing |
| **Hibernate** | RAM saved to EBS, then stop | Preserved + RAM state | No compute charge; EBS billed |
| **Reboot** | Restart | Preserved | Keeps same public IP |

> ⚠️ **Stopping an instance changes its public IP** (unless you use an Elastic IP). Rebooting does not. This is a classic exam gotcha.

### What's lost when you stop?

- **Instance store** (ephemeral) data → **gone**
- **EBS** data → **preserved**
- **Public IPv4** → **released** (new one on restart, unless EIP)

## 6. Pricing Models

How you pay for EC2 — choosing the right one saves real money, and it's heavily tested.

| Model | Description | Best for | Savings |
|---|---|---|---|
| **On-Demand** | Pay per second/hour, no commitment | Short-term, unpredictable, dev/test | Baseline (most expensive) |
| **Reserved Instances (RI)** | Commit to 1 or 3 years | Steady, predictable workloads | Up to ~72% |
| **Savings Plans** | Commit to $/hour spend for 1–3 yrs | Flexible across instance types | Up to ~72% |
| **Spot Instances** | Bid on spare capacity | Fault-tolerant, flexible timing | Up to ~90% |
| **Dedicated Hosts** | Physical server reserved for you | Compliance, licensing (BYOL) | Most expensive |
| **Dedicated Instances** | Hardware not shared, but less control than hosts | Compliance | High |
| **Capacity Reservations** | Reserve capacity in an AZ, no term commitment | Guaranteed capacity needs | No discount |

### Spot Instances — know the details

- Up to **90% cheaper** than On-Demand
- AWS can **reclaim** them with a **2-minute warning** when it needs the capacity
- Perfect for: batch jobs, CI/CD, big data, stateless workloads, anything restartable
- **Never** for: databases, anything that can't tolerate sudden termination

### Reserved Instances — the variations

- **Standard RI** — biggest discount, least flexible
- **Convertible RI** — can change instance family, smaller discount
- **Scheduled RI** — for predictable recurring time windows (legacy)

> 🎯 **Exam decision tree:**
> - Unpredictable, short-term → **On-Demand**
> - Steady 24/7 for years → **Reserved / Savings Plan**
> - Fault-tolerant and cost-sensitive → **Spot**
> - Regulatory/licensing → **Dedicated Host**

## 7. EC2 Storage — EBS and Instance Store

Two fundamentally different kinds of storage attach to EC2.

### EBS (Elastic Block Store)

Network-attached **persistent** block storage. Survives instance stop/terminate (if configured). Think of it as a virtual hard drive.

#### EBS volume types

| Type | Name | Use case |
|---|---|---|
| **gp3 / gp2** | General Purpose SSD | Boot volumes, most workloads (default) |
| **io2 / io1** | Provisioned IOPS SSD | High-performance databases |
| **st1** | Throughput Optimized HDD | Big data, log processing, streaming |
| **sc1** | Cold HDD | Infrequently accessed, lowest cost |

#### Key EBS facts

- **Tied to one Availability Zone** — can't attach across AZs directly
- **Persistent** — survives instance termination if "delete on termination" is off
- **Snapshots** → point-in-time backups, stored in S3, used to create new volumes or move across AZs/regions
- **Encryption** → at rest, in transit, and for snapshots (via KMS)
- Can be **resized** while attached (no downtime in most cases)
- One volume attaches to **one instance** at a time (except **io1/io2 Multi-Attach**)

```
Snapshot a volume → create new volume in another AZ → attach to new instance
(this is how you "move" EBS across AZs)
```

### Instance Store (Ephemeral)

Physically attached local disk on the host machine. **Temporary.**

- **Very fast** (physically attached, often NVMe SSD)
- **Data lost** when the instance stops, terminates, or the host fails
- Survives only a **reboot**
- Good for: caches, scratch data, buffers — never for anything you need to keep

### EBS vs Instance Store

| | EBS | Instance Store |
|---|---|---|
| Persistence | Survives stop/terminate | Lost on stop/terminate |
| Attachment | Network-attached | Physically attached |
| Performance | High (gp3/io2) | Very high |
| Movable | Yes (snapshots) | No |
| Use case | Boot, databases, anything durable | Temp/cache/scratch |

## 8. Security Groups and NACLs

These control network traffic to/from your instances. Both matter — and the difference is a top exam topic.

### Security Groups

A **virtual firewall at the instance level** (technically on the ENI).

- **Stateful** — if you allow inbound traffic, the response is automatically allowed out (and vice versa)
- **Allow rules only** — you cannot create deny rules
- Operate at the **instance/ENI level**
- Evaluated as a **whole** — all rules together
- Can reference **other security groups** (e.g., "allow from the web-tier SG")

Example rules:
```
Inbound:  Allow TCP 22 from 203.0.113.0/24   (SSH from office)
Inbound:  Allow TCP 443 from 0.0.0.0/0        (HTTPS from anywhere)
Outbound: Allow all                            (default)
```

### Network ACLs (NACLs)

A firewall at the **subnet level**.

- **Stateless** — you must explicitly allow both inbound AND outbound (responses aren't auto-allowed)
- **Allow AND deny** rules
- Rules are **numbered** and evaluated in order (lowest first)
- Apply to **all instances in the subnet**

### Security Group vs NACL

| Feature | Security Group | NACL |
|---|---|---|
| Level | Instance / ENI | Subnet |
| State | Stateful | Stateless |
| Rules | Allow only | Allow + Deny |
| Evaluation | All rules together | In number order, stops at first match |
| Default | Deny all inbound, allow all outbound | Allow all (default NACL) |

> 🎯 **Exam favorite:** "Stateful vs stateless." Security Groups remember the connection; NACLs don't. If a NACL allows inbound but you forgot the outbound ephemeral port range, responses get blocked.

## 9. Key Pairs and SSH Access

To log into an instance, you use a **key pair** — public/private key cryptography (recall asymmetric encryption from prerequisites).

### How it works

```
1. AWS stores the PUBLIC key on the instance
2. You keep the PRIVATE key (.pem file)
3. SSH proves you hold the private key → you're let in
```

### Connecting to a Linux instance

```bash
chmod 400 my-key.pem
ssh -i my-key.pem ec2-user@<public-ip>
```

- Default user varies: `ec2-user` (Amazon Linux), `ubuntu` (Ubuntu), `admin` (Debian)
- Windows instances use **RDP** (port 3389) with a decrypted password

### Key pair facts

- **AWS never stores your private key** — if you lose it, you can't recover it
- Lost the key? Options: use EC2 Instance Connect, SSM Session Manager, or detach the volume and fix `authorized_keys` from another instance
- **SSM Session Manager** is the modern, keyless way to access instances (no open SSH port, full audit trail) — increasingly the recommended approach

> 💡 Coming from DevOps: this is the same SSH key model you already know. AWS just manages the public-key injection at launch.

## 10. User Data and Instance Metadata

### User Data — boot-time scripts

A script that runs **once** when the instance first launches. Used to bootstrap the instance — install packages, start services, configure the app.

```bash
#!/bin/bash
yum update -y
yum install -y nginx
systemctl start nginx
systemctl enable nginx
```

- Runs as **root**
- Runs **once** at first boot by default (configurable to run every boot)
- Limited to **16 KB**
- Great for Auto Scaling — every new instance self-configures

### Instance Metadata Service (IMDS)

A special internal endpoint that an instance can query to learn about itself:

```bash
curl http://169.254.169.254/latest/meta-data/
```

Returns things like:
- Instance ID, type, AMI ID
- Public/private IP
- Security groups
- **IAM role credentials** (temporary, auto-rotated)

> ⚠️ **Security note:** Use **IMDSv2** (token-based) instead of IMDSv1. IMDSv1 was vulnerable to SSRF attacks that could steal IAM role credentials through a compromised web app. IMDSv2 requires a session token, blocking that attack.

### User Data vs Metadata

| | User Data | Metadata |
|---|---|---|
| Direction | You → instance (you provide a script) | Instance → itself (it queries info) |
| Purpose | Bootstrap/configure | Self-discovery |
| Endpoint | n/a | `169.254.169.254` |

## 11. Elastic IPs and Networking

### Public IP vs Private IP vs Elastic IP

| Type | Persists across stop/start? | Cost |
|---|---|---|
| **Private IP** | Yes (stays in VPC) | Free |
| **Public IP** | No — changes on stop/start | Free while instance running |
| **Elastic IP (EIP)** | Yes — static, you own it | Free if attached to running instance; charged if idle |

### Elastic IP

A **static public IPv4** you reserve and can remap between instances.

- Stays the same even if the instance stops/starts
- Can be moved to another instance (failover scenarios)
- AWS charges for EIPs **not** associated with a running instance (to discourage hoarding)

> 💡 Modern best practice: avoid EIPs where possible. Use a **load balancer** or **DNS (Route 53)** for a stable endpoint instead.

### Elastic Network Interface (ENI)

A virtual network card. An instance has at least one (`eth0`). You can attach additional ENIs for:
- Multiple IPs / multiple subnets
- Management vs data network separation
- Failover (move an ENI to a standby instance)

## 12. Placement Groups

Control how instances are physically placed to optimize for performance or resilience.

| Type | Layout | Use case |
|---|---|---|
| **Cluster** | Packed close together in one AZ | Low latency, high throughput (HPC, big data) |
| **Spread** | Spread across distinct hardware | Max resilience — critical instances kept apart |
| **Partition** | Grouped into partitions on separate racks | Large distributed systems (HDFS, Cassandra, Kafka) |

```
Cluster:   [I][I][I]  ── same rack, fast network, but shared failure risk
Spread:    [I]  [I]  [I]  ── different hardware, survives hardware failure
Partition: [I][I] | [I][I] | [I][I]  ── isolated racks per partition
```

> 🎯 **Exam mapping:** "lowest latency between instances" → Cluster. "no two instances share hardware" → Spread. "large distributed database" → Partition.

## 13. Auto Scaling and Load Balancing

The combo that makes EC2 truly "elastic."

### Auto Scaling Group (ASG)

Automatically adds or removes instances based on demand or health.

Key settings:
- **Minimum / Desired / Maximum** capacity
- **Launch Template** — defines what new instances look like (AMI, type, SG, user data)
- **Scaling policies** — when to scale (CPU > 70%, request count, schedule)
- **Health checks** — replace unhealthy instances automatically

```
Traffic spikes → ASG launches more instances → traffic drops → ASG terminates extras
```

#### Scaling policy types

| Type | Trigger |
|---|---|
| **Target Tracking** | Keep a metric at a target (e.g., CPU at 50%) |
| **Step Scaling** | Add/remove N instances based on alarm severity |
| **Simple Scaling** | Single adjustment per alarm (older) |
| **Scheduled** | Scale at known times (e.g., 9am Monday) |
| **Predictive** | ML forecasts demand and pre-scales |

### Elastic Load Balancer (ELB)

Distributes incoming traffic across multiple instances.

| Type | Layer | Use case |
|---|---|---|
| **ALB** (Application) | 7 (HTTP/HTTPS) | Web apps, routing by URL/host/header, microservices |
| **NLB** (Network) | 4 (TCP/UDP) | Extreme performance, static IP, low latency |
| **GWLB** (Gateway) | 3 | Deploy third-party virtual appliances (firewalls) |
| **CLB** (Classic) | 4 & 7 | Legacy — avoid for new work |

> 💡 **The classic HA architecture:** ALB in front → distributes to → an Auto Scaling Group of EC2 instances → spread across multiple AZs. This survives instance and AZ failures and scales with demand.

## 14. Monitoring

### CloudWatch

The default monitoring service.

- **Basic monitoring** — metrics every 5 minutes (free)
- **Detailed monitoring** — every 1 minute (paid)
- Metrics: CPU utilization, network in/out, disk I/O, status checks

> ⚠️ **Gotcha:** CloudWatch does **not** see memory or disk space usage by default — those require the **CloudWatch Agent** installed on the instance. This is a popular exam trick.

### Status checks

| Check | What it tests |
|---|---|
| **System Status Check** | AWS's underlying hardware/network |
| **Instance Status Check** | Your instance's OS/config |

If a system check fails → AWS infrastructure issue (try stop/start to migrate to new hardware).
If an instance check fails → your problem (OS, network config, etc.).

## 15. EC2 and IAM

Tying back to the previous chapter — this is how EC2 and IAM connect.

### IAM Roles for EC2

Instead of storing access keys on an instance, you attach an **IAM role** via an **instance profile**. The instance assumes the role and gets temporary credentials automatically through IMDS.

```
EC2 instance → assumes IAM role → temporary creds via IMDS → calls S3/DynamoDB/etc.
```

- **No long-term credentials** stored anywhere
- Credentials **auto-rotate**
- Change permissions by editing the role — no need to touch the instance

> 🎯 This is **the** recommended pattern and a guaranteed interview question: *"How does an EC2 access S3 without hardcoded keys?"* → IAM role + instance profile.

## 16. Best Practices

1. **Use IAM roles, never hardcoded access keys** on instances
2. **Use IMDSv2** to protect role credentials from SSRF attacks
3. **Right-size instances** — don't over-provision; use Compute Optimizer
4. **Use Auto Scaling** for resilience and cost efficiency
5. **Spread across multiple AZs** for high availability
6. **Use Spot for fault-tolerant workloads** to cut costs
7. **Tag everything** — for cost tracking, automation, and organization
8. **Encrypt EBS volumes** and snapshots
9. **Keep AMIs patched** — bake golden images, rotate regularly
10. **Use SSM Session Manager** instead of opening SSH ports
11. **Restrict Security Groups** — least privilege for network too
12. **Enable termination protection** on critical instances
13. **Monitor with CloudWatch + alarms**; install the agent for memory/disk

## 17. Common Scenarios

### Scenario 1: Web app that handles variable traffic
✅ ALB + Auto Scaling Group across multiple AZs, On-Demand or Savings Plan base + Spot for overflow.

### Scenario 2: A nightly batch job that can be interrupted
✅ Spot Instances — up to 90% cheaper, and interruption is acceptable.

### Scenario 3: EC2 needs to read from an S3 bucket
✅ Attach an IAM role (instance profile) with S3 read permission. No keys on the instance.

### Scenario 4: You stopped an instance and now can't reach it on the old IP
✅ Public IP changed on stop/start. Use an Elastic IP, or better, put it behind a load balancer / use Route 53.

### Scenario 5: You need the lowest possible latency between cluster nodes
✅ Cluster placement group in a single AZ.

### Scenario 6: A 24/7 production database server running for 3 years
✅ Reserved Instance or Savings Plan for the steady baseline — big savings over On-Demand.

### Scenario 7: You lost the .pem key for a running Linux instance
✅ Use EC2 Instance Connect or SSM Session Manager; or detach the root volume, attach to another instance, fix `authorized_keys`, reattach.

### Scenario 8: CloudWatch shows CPU but you need memory usage
✅ Install the CloudWatch Agent — memory and disk aren't reported by default.

### Scenario 9: You need instances on separate physical hardware to survive a hardware failure
✅ Spread placement group.

### Scenario 10: You must move an EBS volume from one AZ to another
✅ Snapshot it → create a new volume from the snapshot in the target AZ → attach.
