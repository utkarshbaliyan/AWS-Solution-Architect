## 1. Overview & Core Concepts

### What S3 actually is

Amazon S3 is a **fully managed object storage service**. "Fully managed" means you never think about disks, RAID arrays, replication hardware, or capacity planning — you PUT an object and AWS stores it redundantly across multiple facilities. There is no upper limit on total storage; a bucket can hold trillions of objects.

The headline number is **99.999999999% durability (11 nines)**. To understand what that means concretely: if you store 10 million objects, you can statistically expect to lose **one object every 10,000 years**. AWS achieves this by automatically storing copies of every object across **a minimum of three Availability Zones** (geographically separated data centers) in the Region. This is why losing data is astronomically unlikely.

### Object storage vs block vs file — why it matters

This is a foundational distinction the exam loves:

| Type | AWS service | How data is accessed | Analogy |
|---|---|---|---|
| **Object** | S3 | Whole object via HTTP API (GET/PUT) by key | Valet parking — hand over the whole car, get a ticket |
| **Block** | EBS | OS reads/writes raw blocks; needs a filesystem on top | A raw hard drive you format yourself |
| **File** | EFS / FSx | Mounted filesystem with directories, accessed via NFS/SMB | A shared network drive |

The critical consequence: **you cannot edit part of an S3 object in place.** If you change one byte of a 1 GB file, you re-upload the entire 1 GB object (a full overwrite that creates a new version). This is *object* storage — objects are atomic units. Block storage (EBS) lets you modify individual blocks, which is why databases run on EBS, not S3.

### Anatomy of an object

An S3 object is more than just the file bytes. It consists of:

- **Key** — the full name/path, e.g. `images/2024/logo.png`. This is the unique identifier within a bucket.
- **Value** — the actual data (the bytes), 0 bytes up to 5 TB.
- **Version ID** — present if versioning is enabled; identifies which version this is.
- **Metadata** — key-value pairs. Two flavors:
  - *System metadata* (Content-Type, Content-Length, Last-Modified — managed by S3).
  - *User-defined metadata* (custom headers you set, prefixed `x-amz-meta-`).
- **Tags** — up to 10 key-value tags, used for lifecycle rules, access control, and cost allocation.
- **Access Control** — ACLs / policies governing who can touch it.

### The "folder" illusion

S3 has a **flat namespace**. There are no real directories. When you see `images/logo.png`, the entire string is the key — the `/` is just a character. The console *renders* prefixes as folders for human convenience, but under the hood S3 is a giant key-value store mapping keys to objects.

Why this matters: **prefixes** (the part of the key before a delimiter) affect performance and are how you scope lifecycle rules, replication, and IAM policies. S3 can handle **3,500 PUT/COPY/POST/DELETE and 5,500 GET/HEAD requests per second per prefix**, and this scales horizontally — spreading objects across many prefixes multiplies throughput.

```
AWS Account
└── Region (us-east-1)              <- chosen at creation, permanent
    └── Bucket (my-app-data)         <- globally unique name across ALL AWS accounts
        ├── images/logo.png          <- key = "images/logo.png" (the slash is cosmetic)
        ├── images/banner.jpg        <- shares the "images/" prefix
        └── docs/report.pdf          <- different prefix "docs/"
```

### Size limits to memorize

- **Max object size: 5 TB.**
- **Largest single PUT (one upload operation): 5 GB.** Anything between 5 GB and 5 TB *must* use multipart upload.
- **No limit on number of objects or total bucket size.**

> 📌 **EXAM TIP:** Durability ≠ Availability. **Durability** (11 nines, fixed across almost all classes) = "your data won't be lost." **Availability** = "you can retrieve it right now" and *varies by storage class* (S3 Standard = 99.99%, One Zone-IA = 99.5%). A class can be extremely durable yet temporarily unavailable if an AZ is down.

## 2. Buckets: Structure & Default Settings

### Bucket naming rules (and why they exist)

Bucket names live in a **single global namespace shared by every AWS account on Earth.** If someone has taken `data`, you can never use it. This is because S3 bucket names can become DNS hostnames (`my-bucket.s3.amazonaws.com`), so they must obey DNS rules:

- **3–63 characters.**
- **Lowercase letters, numbers, hyphens, and dots only.** No uppercase, no underscores.
- Must **begin and end with a letter or number.**
- **Cannot be formatted as an IP address** (e.g., `192.168.1.1`).
- Avoid dots if you'll use TLS/virtual-hosted URLs (dots break the wildcard SSL cert).

### Default settings — and what's changeable

When you create a bucket today, AWS applies **secure-by-default** settings. This is a meaningful shift from years ago when buckets were open by default and caused famous data leaks.

| Setting | Default | Changeable? | Notes |
|---|---|---|---|
| **Block Public Access** | **ON — all four blocks enabled** | Yes | Master safety switch; blocks public grants even if a policy allows them |
| **Default Encryption** | **SSE-S3 (AES-256) enabled** | Yes | Since Jan 2023, every new object is encrypted at rest automatically |
| **Versioning** | Disabled | Yes — enable; can only **suspend**, never fully disable afterward | |
| **ACLs** | Disabled ("Bucket owner enforced") | Yes | AWS now recommends keeping ACLs off and using policies |
| **Object Ownership** | Bucket owner enforced | Yes | Owner automatically owns all objects, even cross-account uploads |
| **Region** | Set at creation | **No — permanent** | To "move" a bucket you create a new one and replicate/copy |
| **Bucket name** | Set at creation | **No — permanent** | Cannot rename; must recreate |

### Why "permanent" matters

Region and name being immutable is a common exam trap. If you need a bucket in a different Region, or with a different name, you create a *new* bucket and migrate the data (e.g., via `aws s3 sync`, replication, or S3 Batch Operations). There is no rename or move button.

### The four Block Public Access toggles

"Block Public Access" is actually four independent settings — knowing them shows depth:

1. **BlockPublicAcls** — reject new public ACLs on PUT.
2. **IgnorePublicAcls** — ignore any existing public ACLs.
3. **BlockPublicPolicy** — reject bucket policies that grant public access.
4. **RestrictPublicBuckets** — restrict access to only AWS principals even if a policy is public.

> 📌 **EXAM TIP:** Since early 2023, all new buckets **block public access AND encrypt by default**. Region and name are **immutable**. If a question says "the team accidentally made data public," the answer often involves re-enabling Block Public Access at the account level (which overrides everything below it).

## 3. Access Control: Private vs Public

S3 access is governed by **multiple layers that are evaluated together.** Understanding the evaluation order is what separates a memorizer from someone who understands S3.

### The evaluation pipeline

```
Request arrives
      |
      v
[1] Is there an explicit DENY anywhere? --> YES -> DENY (deny always wins)
      | NO
      v
[2] Block Public Access blocking it? --> YES -> DENY
      | NO
      v
[3] Does IAM / Bucket policy / ACL ALLOW it? --> NO -> DENY (implicit deny)
      | YES
      v
   ALLOW
```

Two rules drive everything:
- **Default deny:** if nothing explicitly allows the action, it's denied.
- **Explicit deny wins:** a single `Deny` anywhere overrides any number of `Allow`s.

### The four access mechanisms

**1. IAM Policies (identity-based)**
Attached to IAM users, groups, or roles. They say *"this principal can do these S3 actions on these resources."* Best when controlling what *your own account's* users can do. The principal is implied (it's whoever the policy is attached to).

**2. Bucket Policies (resource-based)**
A JSON document attached directly to the bucket. They say *"these principals can do these actions on this bucket."* Because they explicitly name a `Principal`, they are the standard tool for:
- Granting **public** access (`"Principal": "*"`).
- Granting **cross-account** access.
- Enforcing conditions (force HTTPS, require encryption, restrict by IP/VPC).

**3. ACLs (Access Control Lists — legacy)**
Coarse-grained, predates IAM. Can grant access per-object or per-bucket to predefined groups. **AWS now recommends disabling ACLs entirely** ("Bucket owner enforced"). Mentioned here because the exam still references them, but in modern design you use policies.

**4. Block Public Access (BPA)**
The master override discussed above. Sits *above* policies and ACLs — if BPA blocks public access, it doesn't matter what your bucket policy says.

### Making a bucket public (the static-website case)

To serve, say, a static website's assets publicly:

1. **Turn OFF Block Public Access** (uncheck the relevant boxes).
2. **Add a bucket policy** granting `s3:GetObject` to everyone:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "PublicReadGetObject",
      "Effect": "Allow",
      "Principal": "*",
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::my-bucket/*"
    }
  ]
}
```

Note `Resource` ends in `/*` (applies to *objects inside* the bucket), not the bucket ARN itself.

### The secure alternative (what AWS actually wants you to do)

Making a bucket public is rarely the right answer in production. The recommended pattern is:

- Keep the bucket **fully private** (Block Public Access ON).
- Put **CloudFront** (a CDN) in front of it.
- Use an **Origin Access Control (OAC)** so only CloudFront can read from the bucket.

This gives you HTTPS, caching, lower latency, DDoS protection, and the bucket is never directly exposed.

### Useful condition-based policies

Force HTTPS (deny any non-TLS request):

```json
{
  "Effect": "Deny",
  "Principal": "*",
  "Action": "s3:*",
  "Resource": ["arn:aws:s3:::my-bucket", "arn:aws:s3:::my-bucket/*"],
  "Condition": { "Bool": { "aws:SecureTransport": "false" } }
}
```

> 📌 **EXAM TIP:** **Explicit Deny always wins** over any Allow. **Block Public Access overrides bucket policies.** For public-facing content the *correct, secure* answer is almost always **CloudFront + OAC with a private bucket**, not a public bucket policy.

## 4. Versioning & Delete Markers

### The problem versioning solves

Without versioning, S3 keys are unique — uploading `report.pdf` again **overwrites** the old one permanently, and deleting it is gone forever. Versioning protects against both **accidental overwrites** and **accidental deletes**.

### How it works

Enable versioning **at the bucket level**. Once on, every PUT to the same key creates a **new version** with a unique version ID; old versions are retained. You can list, retrieve, or restore any prior version.

Key mechanics:
- Versioning is **enabled once and can only be *suspended*, never turned fully off.** Suspending stops creating *new* versions but keeps existing ones.
- Objects that existed **before** you enabled versioning have a version ID of **`null`**.
- Each version is billed as a separate object — old versions **cost money**, so pair versioning with lifecycle rules to expire noncurrent versions.

### Delete markers — the subtle part

This trips up many learners. When you **delete a versioned object without specifying a version ID**, S3 does *not* erase anything. Instead it places a **delete marker** as the newest version — essentially a tombstone with no data.

Consequences:
- A subsequent **GET returns HTTP 404** (S3 sees the delete marker on top and behaves as if the object is gone).
- The actual data versions still exist underneath, still costing storage.
- **To restore the object:** delete the *delete marker* (specify its version ID). The previous real version becomes current again — instant undo.
- **To permanently delete a specific version:** issue a delete that *specifies the version ID*. This is the only way to actually remove data and reclaim storage.

```
Timeline:
  PUT report.pdf   -> creates v1 (data)
  PUT report.pdf   -> creates v2 (data), v1 retained
  DELETE report.pdf (no version) -> adds [delete marker] as latest

Bucket now contains:  v1, v2, [delete marker <- current]
  GET report.pdf   -> 404 (delete marker is on top)

  DELETE the delete marker -> v2 becomes current again
  GET report.pdf   -> returns v2  (restored)
```

### MFA Delete — extra protection

You can enable **MFA Delete** on a versioned bucket. This requires a valid MFA token to:
- Permanently delete a specific object version, or
- Suspend versioning on the bucket.

It can only be enabled by the **bucket owner (root account)** via CLI, and adds a strong guard against malicious or accidental destruction.

> 📌 **EXAM TIP:** Deleting a versioned object inserts a delete marker (GET -> 404) but doesn't remove data. Only a **version-ID-specific delete** permanently removes data. Combine **versioning + MFA Delete** when a question stresses protecting against accidental *or malicious* deletion.

## 5. Uploads: Size Limits & Multipart

### The numbers

| Limit | Value |
|---|---|
| Max object size | 5 TB |
| Max size in a **single** PUT | 5 GB |
| Multipart **recommended** above | 100 MB |
| Multipart **required** above | 5 GB |
| Parts per upload | 1 – 10,000 |
| Part size range | 5 MB – 5 GB (the **last** part may be smaller than 5 MB) |

### Why single PUT caps at 5 GB

A single HTTP PUT is one network stream. If it fails at 4.9 GB, you start over — wasteful and fragile over real-world networks. So for anything large, S3 offers **multipart upload**.

### How multipart upload works

The object is split into independent **parts**, each uploaded **separately and in parallel**, then S3 **reassembles** them server-side into one object.

The three-phase flow:
1. **Initiate** — call `CreateMultipartUpload`; S3 returns an `UploadId`.
2. **Upload parts** — upload each part (numbered 1–10,000) referencing the `UploadId`. Each returns an **ETag**. Parts can go up in parallel and be retried individually.
3. **Complete** — call `CompleteMultipartUpload` with the list of part numbers + ETags; S3 stitches them into the final object. (Or `AbortMultipartUpload` to cancel.)

Benefits:
- **Parallelism** -> much faster for large files.
- **Resilience** -> a failed part is retried alone, not the whole file.
- **Pause/resume** -> uploads can span time.

### The hidden cost trap

If an upload is initiated but never completed or aborted, the **uploaded parts sit in the bucket consuming storage you pay for** — and they're invisible in the normal object listing. This silently wastes money.

**Fix:** add a lifecycle rule to **abort incomplete multipart uploads after N days** (e.g., 7). This is a best-practice almost every well-architected bucket should have.

```bash
# CLI handles multipart automatically for large files:
aws s3 cp ./huge-video.mp4 s3://my-bucket/   # SDK/CLI auto-splits if large
```

> 📌 **EXAM TIP:** Multipart is **required above 5 GB**, recommended above 100 MB. Always pair it with a lifecycle rule to **abort incomplete multipart uploads**, and with **Transfer Acceleration** when uploading large files over long distances.

## 6. Storage Classes

Storage classes let you trade **access speed and availability** for **lower cost**. Durability is **11 nines across all classes** (One Zone is still 11 nines but within a single AZ). What changes is availability, retrieval time, retrieval cost, and minimum billing duration.

### The full lineup

| Class | Best for | Min duration | Retrieval time | AZs | Relative cost |
|---|---|---|---|---|---|
| **S3 Standard** | Hot, frequently accessed | none | milliseconds | ≥3 | highest |
| **S3 Intelligent-Tiering** | Unknown/changing access | none | milliseconds | ≥3 | high + monitoring fee |
| **S3 Standard-IA** | Infrequent but needs fast access | 30 days | milliseconds | ≥3 | medium + retrieval fee |
| **S3 One Zone-IA** | Infrequent + re-creatable | 30 days | milliseconds | **1** | low + retrieval fee |
| **S3 Glacier Instant Retrieval** | Archive needing instant access | 90 days | milliseconds | ≥3 | lower + retrieval fee |
| **S3 Glacier Flexible Retrieval** | Archive, rare access | 90 days | minutes–hours | ≥3 | very low |
| **S3 Glacier Deep Archive** | Long-term cold archive | 180 days | 12–48 hours | ≥3 | cheapest |

### How to reason about each

**S3 Standard** — the default. Low latency, high throughput, 99.99% availability. Use for active workloads: websites, content distribution, data analytics input.

**S3 Intelligent-Tiering** — you don't know or can't predict the access pattern. S3 **automatically moves each object** between a frequent-access tier and infrequent/archive tiers based on observed usage. You pay a small **per-object monthly monitoring fee** but **no retrieval fees**. The catch: the monitoring fee makes it inefficient for huge numbers of *tiny* objects. Ideal when access is genuinely unpredictable.

**S3 Standard-IA (Infrequent Access)** — data accessed rarely but that must come back **fast** when needed (e.g., monthly reports, backups you occasionally restore). Cheaper storage than Standard, but you pay a **per-GB retrieval fee** and there's a **30-day minimum** — delete before 30 days and you're still billed for 30.

**S3 One Zone-IA** — same as Standard-IA but stored in **a single AZ**, making it ~20% cheaper. The trade-off: **if that AZ is destroyed, the data is gone.** Only use for data you can **regenerate** (thumbnails, transcoded media, secondary copies).

**Glacier Instant Retrieval** — archive pricing with **millisecond** retrieval. For data accessed maybe once a quarter but needed instantly when it is (medical images, news archives).

**Glacier Flexible Retrieval** — cheaper archive where you can **wait** for retrieval. Three retrieval speeds:
- **Expedited:** 1–5 minutes
- **Standard:** 3–5 hours
- **Bulk:** 5–12 hours (cheapest)

**Glacier Deep Archive** — the **cheapest storage in all of AWS.** Retrieval takes **12–48 hours**. For compliance data and long-term archives you almost never read (7-year financial records, legal holds). 180-day minimum.

### Choosing fast

- "Lowest possible cost, hours-to-retrieve is fine" -> **Deep Archive**.
- "Access pattern unknown / changes over time" -> **Intelligent-Tiering**.
- "Infrequent but re-creatable, save money" -> **One Zone-IA**.
- "Archive but I need it back in milliseconds" -> **Glacier Instant Retrieval**.

> 📌 **EXAM TIP:** Watch for **minimum-duration penalties** (IA = 30d, Glacier = 90d, Deep Archive = 180d). Watch for the **single-AZ risk** of One Zone-IA. "Unknown access pattern" is almost always **Intelligent-Tiering**.

## 7. Lifecycle Management

### The idea

Manually moving objects between storage classes at scale is impossible. **Lifecycle policies** automate it — rules that **transition** objects to cheaper classes and **expire** (delete) them on a schedule, all based on object age.

### The two action types

**1. Transition actions** — move objects to a cheaper storage class after N days.
Example progression for log data:

```
Day 0:    S3 Standard           (logs are hot, queried often)
Day 30:   -> S3 Standard-IA      (rarely queried now)
Day 90:   -> S3 Glacier Flexible (archival)
Day 365:  -> S3 Glacier Deep Archive
```

**2. Expiration actions** — permanently delete objects after N days. Expiration can also:
- Delete **noncurrent (old) versions** after N days — essential for controlling versioning costs.
- **Expire delete markers** that no longer have associated versions (cleanup).
- **Abort incomplete multipart uploads** after N days (the cost-saver from section 5).

```
Day 2555 (7 years): Expire current version -> object deleted
Noncurrent versions: delete 30 days after becoming noncurrent
Incomplete multipart uploads: abort after 7 days
```

### Scoping rules

Rules can apply to:
- The **whole bucket**, or
- A **prefix** (e.g., only `logs/`), or
- Objects with specific **tags** (e.g., `Project=Archive`).

This lets one bucket have different lifecycle behavior per data category.

### Important constraints (exam gold)

- **Transitions only go "downhill"** to colder/cheaper classes. You **cannot** lifecycle-transition Glacier -> Standard. Getting data *out* of Glacier requires an explicit **restore** operation (which creates a temporary copy), not a lifecycle rule.
- **Minimum-duration penalties still apply.** A lifecycle rule moving objects to Standard-IA on day 10 then deleting on day 20 still bills the **30-day minimum** for Standard-IA.
- There's a **minimum transition timing**: objects generally must wait before transitioning to certain colder classes (S3 enforces sensible minimums so you don't thrash).

> 📌 **EXAM TIP:** Lifecycle = transition + expiration. Use it to **expire old versions** (versioning cost control) and **abort incomplete multipart uploads** (hidden cost control). Remember transitions are one-way (downhill); restoring from Glacier is a separate restore operation, not a lifecycle transition.

## 8. Logging: Server Access & Object-Level

There are **two distinct logging systems** in S3, and the exam wants you to know which to use when.

### Server Access Logging

Logs **every request** made to the bucket — requester, bucket name, request time, action, response status, error codes, bytes transferred.

- Logs are **written to a separate target S3 bucket** (it cannot be the same bucket, or you'd create an infinite logging loop).
- **Best-effort delivery** — logs may arrive hours late and occasionally be incomplete. Not for real-time alerting.
- **Free** feature (you only pay storage for the logs).
- Format is a fixed space-delimited text — good for usage analysis and access patterns, less convenient for security forensics.

### Object-Level Logging (CloudTrail Data Events)

CloudTrail can record **API-level operations on individual objects** — `GetObject`, `PutObject`, `DeleteObject`, etc. — capturing *who* (which IAM principal) did *what* and *when*.

- Delivered to **CloudTrail** (and onward to CloudWatch Logs, S3, EventBridge).
- **Near real-time**, structured (JSON), and integrates with security tooling.
- **Paid** (data events cost extra per event).
- Designed for **security auditing, compliance, and forensics.**

### Side-by-side

| | Server Access Logging | CloudTrail Data Events |
|---|---|---|
| Granularity | Per HTTP request | Per API call on an object |
| Latency | Hours (best-effort) | Near real-time |
| Format | Space-delimited text | Structured JSON |
| Cost | Free | Paid |
| Captures IAM identity clearly | Limited | Yes |
| Best for | Usage/traffic analysis | Security & compliance auditing |

> 📌 **EXAM TIP:** "Need to know **which user** deleted/accessed an object, in near real-time, for security/compliance" -> **CloudTrail data events (object-level logging)**, NOT server access logs. Server access logs are for broad traffic/usage analysis.

## 9. CORS (Cross-Origin Resource Sharing)

### The browser rule it works around

Browsers enforce the **same-origin policy**: a web page can only make requests to its own origin by default. An **origin** = `scheme + host + port` (e.g., `https://app.example.com:443`). If a page at one origin tries to fetch a resource from a *different* origin, the browser blocks it unless the resource's server explicitly opts in via **CORS headers**.

### When this hits S3

Imagine your single-page app is hosted at `https://app.example.com` but loads images, fonts, or makes JavaScript `fetch()` calls to assets in an S3 bucket served from `https://my-assets.s3.amazonaws.com`. Those are **different origins**. Without CORS configured on the bucket, the browser refuses the cross-origin request and you see a CORS error in the console.

### The fix — CORS configuration on the bucket

You attach a CORS configuration to the **bucket whose resources are being requested** (the *resource* bucket, not the page's bucket):

```json
[
  {
    "AllowedOrigins": ["https://app.example.com"],
    "AllowedMethods": ["GET", "PUT", "POST"],
    "AllowedHeaders": ["*"],
    "ExposeHeaders": ["ETag"],
    "MaxAgeSeconds": 3000
  }
]
```

- **AllowedOrigins** — which origins may make cross-origin requests (`*` for any, or specific domains).
- **AllowedMethods** — HTTP methods permitted (GET, PUT, POST, DELETE, HEAD).
- **AllowedHeaders** — which request headers the browser may send.
- **ExposeHeaders** — which response headers JS is allowed to read.
- **MaxAgeSeconds** — how long the browser caches the preflight result.

S3 responds to the browser's **preflight `OPTIONS` request** with the matching headers, and the browser then permits the real request.

> 📌 **EXAM TIP:** CORS errors when a website loads assets/fonts from a *different* S3 bucket -> add a **CORS configuration to the bucket being requested** (the resource bucket). It's set on the resource side, not the requester side.

## 10. Replication: CRR & SRR

### What replication does

S3 Replication **asynchronously copies objects** from a source bucket to one or more destination buckets, automatically, as objects are written.

### The two flavors

| | **CRR — Cross-Region Replication** | **SRR — Same-Region Replication** |
|---|---|---|
| Destination | A bucket in a **different Region** | A bucket in the **same Region** |
| Primary use cases | Disaster recovery, lower-latency access for distant users, meeting data-residency/compliance rules | Aggregating logs from multiple buckets, syncing prod/test, in-region data sovereignty |

### Hard requirements (commonly tested)

1. **Versioning must be enabled on BOTH source and destination buckets.** Replication is built on versioning.
2. An **IAM role** granting S3 permission to read the source and write the destination.
3. The S3 service must be allowed to replicate (role + bucket policies aligned).

### Critical behaviors

- **Asynchronous** — replication isn't instant; objects appear in the destination shortly after.
- **Only NEW objects replicate by default.** Objects that existed *before* replication was turned on are **not** copied automatically — use **S3 Batch Replication** to backfill existing objects.
- **No chaining.** If A replicates to B, and B replicates to C, objects from A do **not** automatically reach C. Each replication is a direct source->destination relationship.
- **Cross-account replication** is supported — the destination can be in a different AWS account, useful for isolating backups from a potentially compromised primary account.
- **Delete handling (important nuance):**
  - **Delete markers CAN optionally be replicated** (configurable).
  - **Deletes that specify a version ID are NOT replicated.** This is deliberate — it prevents an attacker (or mistake) who permanently deletes versions in the source from having those permanent deletes propagate and destroy your replica too.

```
Source bucket (us-east-1)  --async-->  Destination bucket (eu-west-1)
   versioning ON                          versioning ON
   new PutObject  ------------------->     replicated copy appears

   DELETE (version-id specified)  --X-->   NOT replicated (replica preserved)
   DELETE marker (optional)       -->-->   replicated if configured
```

> 📌 **EXAM TIP:** Both buckets **need versioning**. Replication is **async** and only affects **new** objects (use Batch Replication for old ones). **Version-ID deletes do not replicate** — a key safety feature. CRR = different Region (DR/latency/compliance); SRR = same Region (log aggregation/prod-test).

## 11. Encryption

S3 encryption splits into **at rest** (stored on disk) and **in transit** (moving over the network). At rest further splits into **server-side** (S3 encrypts/decrypts for you) and **client-side** (you do it before upload).

### Encryption at rest — Server-Side Encryption (SSE)

| Type | Who manages the key | How it works | When to choose |
|---|---|---|---|
| **SSE-S3** | AWS, fully | S3 owns and rotates the keys; transparent (AES-256). Default on all new buckets. | Simplest; you just want "encrypted at rest" with zero management |
| **SSE-KMS** | AWS KMS (you control the key policy) | Uses a KMS key; every encrypt/decrypt is **logged in CloudTrail**; supports key rotation and fine-grained access control | You need an **audit trail of key usage**, key rotation, and access control over who can decrypt |
| **SSE-C** | **You** (customer-provided) | You send the encryption key with every request over HTTPS; **AWS uses it then discards it** — never stored | Compliance requires that **AWS never holds your keys**, but you still want S3 to do the encryption work |
| **DSSE-KMS** | AWS KMS (double layer) | Applies **two independent layers** of KMS encryption | Strict regulatory environments mandating double encryption |

### Encryption at rest — Client-Side Encryption

You encrypt the data **on your own machine before uploading**. S3 only ever receives and stores ciphertext; AWS cannot read it. You are fully responsible for the keys and the encryption/decryption logic (often via the AWS Encryption SDK). Maximum control, maximum responsibility.

### Encryption in transit

Protects data **moving to and from S3** using **TLS/SSL (HTTPS)**. S3 supports both HTTP and HTTPS endpoints; to *require* encryption in transit, deny non-TLS requests with a bucket policy condition:

```json
"Condition": { "Bool": { "aws:SecureTransport": "false" } }
```

(used in a `Deny` statement — see section 3).

### The algorithm

All SSE options encrypt object data using **AES-256** — the **Advanced Encryption Standard with a 256-bit key**, a symmetric block cipher and the industry standard for data at rest. SSE-KMS and DSSE-KMS layer KMS key management on top of AES-256.

### Choosing fast

- "Simplest, just encrypt it" -> **SSE-S3**.
- "Audit who decrypted, control access, rotate keys" -> **SSE-KMS**.
- "AWS must never store our keys" -> **SSE-C** (S3 still encrypts) or **client-side** (S3 never sees plaintext at all).
- "Double encryption for compliance" -> **DSSE-KMS**.

> 📌 **EXAM TIP:** **SSE-KMS** = audit trail + key control (the most common "compliance" answer). **SSE-C** = customer supplies key, AWS doesn't store it. **Client-side** = AWS never sees plaintext. Algorithm is **AES-256**. Force HTTPS via the `aws:SecureTransport` condition.

## 12. Consistency Model

### What "consistency" means here

Consistency answers: *after I write data, when I immediately read it, do I get the latest version?* A **strongly consistent** system always returns the most recent write. An **eventually consistent** system might briefly return stale data.

### S3 today

Since **December 2020**, S3 provides **strong read-after-write consistency** for **all operations, automatically, at no extra cost and no performance penalty.**

Concretely:
- **PUT a brand-new object**, then immediately **GET** it -> you get the object (no 404 race).
- **Overwrite** an existing object (PUT), then **GET** -> you get the new version, never the old.
- **DELETE**, then **GET/LIST** -> reflects the deletion immediately.
- **LIST** operations immediately reflect added/removed objects.

This applies across GET, PUT, LIST, and metadata operations.

### Why this matters for the exam

Older study materials, courses, and exam dumps still say *"S3 is eventually consistent for overwrite PUTs and DELETEs, strongly consistent for new-object PUTs."* **That is outdated.** S3 is now **strongly consistent for everything.** If you see a question or answer implying eventual consistency for overwrites/deletes, it reflects pre-2020 behavior.

> 📌 **EXAM TIP:** S3 is **strongly read-after-write consistent for all operations** (since Dec 2020). Ignore older "eventual consistency" claims — they're obsolete.

## 13. Pre-Signed URLs

### The problem

You have **private** objects (bucket not public) but want to give a specific person **temporary** access — e.g., "download your invoice" or "let a user upload a file directly" — **without making the bucket public and without sharing AWS credentials.**

### How pre-signed URLs work

A pre-signed URL is a regular S3 object URL with **authentication baked into the query string** and an **expiration time**. It's generated by an entity (user, role, or app) that already has permission to the object. Anyone holding the URL can perform the allowed action **until it expires** — no AWS account needed.

Crucially: **the URL carries the permissions of whoever generated it.** If a Lambda with read access to `invoices/` generates a pre-signed GET URL, the recipient can read that object as if they were that Lambda — but only that object, only that action, only until expiry.

### Two directions

- **Pre-signed GET** — temporary **download** of a private object ("here's a 15-minute link to your report").
- **Pre-signed PUT** — temporary **upload** directly to S3 ("upload your profile photo straight to our bucket"), so files never pass through your server and the bucket stays private.

### Expiration

- Generated via SDK or CLI.
- CLI default expiry is **3,600 seconds (1 hour)**; configurable.
- With SigV4, max validity is **7 days**.

```bash
# Generate a 1-hour download link for a private object:
aws s3 presign s3://my-bucket/invoices/inv-2024.pdf --expires-in 3600
```

### Classic use cases

- Paid content / premium downloads with expiring links.
- Direct browser-to-S3 uploads (offload bandwidth from your app servers).
- Sharing a single file with someone who has no AWS account.

> 📌 **EXAM TIP:** "Allow users to **upload/download a private object** without making the bucket public and without giving them credentials" -> **pre-signed URL** (PUT for upload, GET for download). The URL inherits the **generator's permissions** and **expires**.

## 14. Transfer Acceleration

### The problem it solves

When a user in, say, India uploads a large file to a bucket in `us-east-1`, the data crosses the public internet over many hops — slow and variable, especially for large files over long distances.

### How it works

**S3 Transfer Acceleration** routes the upload to the **nearest CloudFront edge location** (Amazon has edge locations worldwide), then carries the data over **AWS's private, optimized backbone network** to the destination bucket — avoiding much of the slow public-internet path.

- Uses a **distinct endpoint**: `my-bucket.s3-accelerate.amazonaws.com`.
- Most beneficial for **large files** transferred over **long geographic distances**. For small files or nearby clients, the benefit is marginal.
- **Costs extra** per GB transferred.
- AWS provides a **speed comparison tool** so you can measure whether acceleration actually helps for your route before committing.

### Pairs well with

Combine with **multipart upload** for the fastest possible large, long-distance uploads — multipart parallelizes, Transfer Acceleration optimizes the network path.

```
User (Mumbai) --> nearest edge location --> AWS private backbone --> bucket (us-east-1)
                    (fast on-ramp)             (optimized path)
```

> 📌 **EXAM TIP:** Large files + distant users + need faster uploads -> **Transfer Acceleration** (edge + AWS backbone). Combine with **multipart upload** for maximum throughput.

## 15. Requester Pays

### Normal billing vs Requester Pays

Normally the **bucket owner pays for everything**: storage, requests, and data transfer (egress). If you host a large public dataset, every download costs *you* money — potentially enormous bandwidth bills.

**Requester Pays** flips part of this: with it enabled, the **person making the request pays for the request and data-transfer costs**, while the **bucket owner continues to pay only for storage.**

### How it works

- The requester must be an **authenticated AWS user** (anonymous/public access is not allowed in Requester Pays buckets — the requester needs an AWS account to bill).
- Requests must explicitly include the header `x-amz-request-payer: requester` to acknowledge they accept the charges.
- Common for **sharing large datasets** (scientific data, ML training sets) where the owner is willing to host but not to absorb everyone's download bandwidth.

> 📌 **EXAM TIP:** "Share a large dataset publicly **without paying the download/bandwidth costs**" -> **Requester Pays**. Requesters must be **authenticated** AWS users (no anonymous access).

## 16. Event Notifications

### The idea

S3 can **emit events when things happen to objects** and route them to other AWS services to trigger automated workflows — turning S3 into an event source for serverless and decoupled architectures.

### What triggers events

Object-level actions, including:
- `s3:ObjectCreated:*` (Put, Post, Copy, CompleteMultipartUpload)
- `s3:ObjectRemoved:*` (delete, delete marker created)
- `s3:ObjectRestore:*` (Glacier restore initiated/completed)
- `s3:Replication:*` and others.

You can filter by **prefix** and **suffix** (e.g., only `uploads/` objects ending in `.jpg`).

### Where events can go (destinations)

| Destination | Use it for |
|---|---|
| **SNS** | Fan-out a notification to many subscribers (email, other queues, HTTP) |
| **SQS** | Drop the event into a queue for reliable, decoupled, asynchronous processing |
| **Lambda** | Run code immediately (e.g., generate a thumbnail, scan a file, update a DB) |
| **EventBridge** | Advanced filtering on metadata/tags, **many** additional targets, archive & **replay**, cross-account routing |

```
            Object uploaded (s3:ObjectCreated:*)
                          |
        +-------------+---+--------+---------------+
        v             v            v               v
      SNS           SQS         Lambda        EventBridge
   (fan-out)    (queue/buffer) (run code)  (filter/route/replay)
```

### Important mechanics

- Each destination's **resource policy must allow S3 to publish** to it (e.g., the SQS queue policy / Lambda permission must trust S3).
- Delivery is typically within **seconds** but can occasionally be delayed — design consumers to be **idempotent** and tolerant of out-of-order/duplicate events.
- **EventBridge** unlocks far richer filtering (on object metadata and tags) and integrates with many AWS services as targets, plus replay/archive — choose it when simple prefix/suffix filtering isn't enough.

> 📌 **EXAM TIP:** "Process uploaded files automatically" -> S3 Event Notification -> **Lambda** (or **SQS** for buffering/decoupling, **SNS** for fan-out). Need **advanced filtering, many targets, or replay** -> route through **EventBridge**.

## 17. S3 Access Points

### The problem at scale

A single bucket shared by **many applications and teams**, each needing different permissions, leads to **one enormous, fragile bucket policy** that's hard to read, hard to audit, and easy to break. There's also a size limit on bucket policies you can hit.

### What Access Points do

An **S3 Access Point** is a **named network endpoint attached to a bucket, each with its own dedicated access policy.** Instead of cramming every team's rules into the bucket policy, you give each team/app its **own access point** with a focused policy.

Each access point has:
- A **unique DNS name** (its own hostname to address the bucket through).
- Its **own access policy** (scoped to that consumer's needs).
- An optional **VPC restriction** — make the access point reachable **only from a specific VPC**, so that data path never touches the public internet.

```
                     +---------------------+
   Finance app --->  | AP: finance-ap      | --+
   (policy: read     | (own policy, VPC-X) |   |
    finance/* only)  +---------------------+   |
                                               v
                     +---------------------+  +--------------+
   Analytics app -->  | AP: analytics-ap    |->|   Bucket     |
   (policy: read     | (own policy)        |  | shared-data  |
    all, no write)   +---------------------+  +--------------+
```

### Multi-Region Access Points

A **Multi-Region Access Point** provides a **single global endpoint** that automatically routes requests to the **nearest bucket** among several buckets replicated across Regions — simplifying global, low-latency, highly available access behind one hostname.

### Why it's better than a giant bucket policy

- Each consumer's permissions are **isolated and readable**.
- You can **add/remove/modify** one team's access without touching others.
- **VPC-only** access points enforce private networking.
- Scales to hundreds of distinct access patterns on one bucket.

> 📌 **EXAM TIP:** "A large shared bucket, many teams each needing different scoped access, and the bucket policy is becoming unmanageable" -> **S3 Access Points** (one per consumer, each with its own policy, optionally **VPC-restricted**). For global low-latency access to replicated buckets -> **Multi-Region Access Points**.

## 18. Bonus Topics You May Be Missing

These appear on SAA-C03 and complete a strong S3 mental model:

- **S3 Object Lock (WORM — Write Once Read Many)** — prevents an object version from being deleted or overwritten for a fixed retention period or indefinitely (legal hold). Two modes: **Governance** (users with special permission can still override) and **Compliance** (no one — not even the root account — can delete until retention expires). Requires versioning. The go-to answer for "regulatory immutability."

- **S3 Static Website Hosting** — serve HTML/CSS/JS directly from a bucket. Endpoint looks like `bucket.s3-website-<region>.amazonaws.com`. **HTTP only** — for HTTPS you front it with **CloudFront**.

- **S3 Storage Lens** — organization-wide analytics dashboard showing storage usage, activity trends, and cost-optimization recommendations across all buckets and accounts.

- **S3 Inventory** — scheduled (daily/weekly) **CSV/ORC/Parquet reports** listing your objects and their metadata (encryption status, replication status, size). Used for auditing and reconciling at scale instead of expensive LIST calls.

- **S3 Select / Glacier Select** — run **SQL** to retrieve only a **subset** of data from within a single object (CSV/JSON/Parquet), so you transfer and pay for far less than downloading the whole object.

- **S3 Batch Operations** — perform a single action (copy, set tags, restore from Glacier, invoke a Lambda) across **billions of objects** driven by a manifest (often an S3 Inventory report).

- **VPC Gateway Endpoint for S3** — access S3 **privately from a VPC** without an internet gateway or NAT — traffic stays on the AWS network. **Free.** The standard pattern for private EC2 instances reaching S3.

- **CloudFront + OAC (Origin Access Control)** — the recommended way to serve S3 content over **HTTPS with caching and global low latency** while keeping the bucket **private**. (OAC is the modern replacement for the older OAI.)

- **IAM Access Analyzer for S3** — automatically flags buckets and access points that grant **public or cross-account** access, helping you catch unintended exposure.

- **MFA Delete** — requires an MFA token to permanently delete object versions or suspend versioning (see section 4).

- **Bucket Keys** — an SSE-KMS cost optimization that reduces calls to KMS (and thus KMS charges) by using a bucket-level key to encrypt objects.

## 21. Interview Questions (Tiered)

### Tier 1 — Fundamentals
- What is S3, and how does object storage differ from block (EBS) and file (EFS) storage?
- Explain the difference between durability and availability in S3.
- Walk me through what makes up an S3 object (key, value, version ID, metadata, tags).
- How do you make an S3 object publicly accessible — and what's the *secure* alternative?
- What is versioning, and what is a delete marker?

### Tier 2 — Intermediate
- A bucket has mixed hot and cold data and the bill is climbing. How do you cost-optimize it? (Storage classes + Intelligent-Tiering + lifecycle transitions/expiration + expiring old versions + aborting incomplete multipart uploads.)
- Compare SSE-S3, SSE-KMS, SSE-C, and client-side encryption — when would you pick each?
- How does multipart upload improve reliability and performance for large files, and what hidden cost does it introduce?
- Explain CRR vs SRR, their requirements, and a real use case for each.
- A web app throws CORS errors loading fonts from an S3 bucket — diagnose and fix it.
- Explain S3's consistency model and how it changed in 2020.

### Tier 3 — Architecture & Scenario
- Design a flow where **external users upload files directly to S3** without AWS credentials, and each upload **triggers processing**. (Pre-signed PUT URL -> `s3:ObjectCreated` event -> Lambda/SQS; bucket stays private.)
- Architect serving **private media globally over HTTPS with low latency**. (CloudFront + OAC + private bucket; Transfer Acceleration for uploads; appropriate cache behaviors.)
- A single bucket is shared by 10 teams each needing different scoped access, and the bucket policy is now unmanageable — what's your design? (S3 Access Points per team, each with a focused policy, optionally VPC-restricted.)
- Regulatory rule: financial records must be **undeletable for 7 years**, even by admins. How do you implement it? (Object Lock **Compliance** mode + versioning + lifecycle to Deep Archive.)
- You must prove **which IAM principal deleted a specific object** last month — how? (CloudTrail **data events** / object-level logging, not server access logs.)
- Guarantee **all objects are encrypted at rest and in transit, and enforce it** so non-compliant uploads fail. (Default SSE + bucket policy denying `aws:SecureTransport=false` and denying unencrypted PutObject via condition keys.)
- Design a **disaster-recovery** strategy for critical S3 data across Regions and accounts, resilient to malicious deletion. (CRR to a different account/Region, versioning + MFA Delete, deny version deletes from propagating, optional Object Lock on the replica.)
- A private EC2 fleet in a VPC needs S3 access without exposing traffic to the internet. (S3 **Gateway VPC Endpoint** — free, keeps traffic on the AWS backbone.)
